# Simple Document Analysis Pipeline

For a small team, I'd recommend this straightforward approach:

## Architecture Overview

```
PDF Input → Extraction → Processing → Output
              ↓
         Knowledge Base Lookup
```

**Core components:**

1. **PDF Processing Layer**
   - Use `PyPDF2` or `pdfplumber` for text extraction
   - For scanned PDFs, add `pytesseract` (OCR)
   - Extract text + basic structure (sections, tables)

2. **Information Extraction**
   - Start with regex/keyword matching for known patterns
   - Consider `spaCy` for NER (entity recognition) if you need it later
   - Build a simple schema matching your domain

3. **Knowledge Base Lookup**
   - SQLite or PostgreSQL (minimal overhead)
   - Simple key-value store for lookups
   - Start with exact/fuzzy string matching (use `fuzzywuzzy`)

4. **Output Generation**
   - Structured format: JSON or YAML
   - Template-based summaries (Jinja2)

## Minimal Working Example

```python

import json
from fuzzywuzzy import fuzz
import sqlite3

def extract_text(pdf_path):
    text = ""
    with pdfplumber.open(pdf_path) as pdf:
        for page in pdf.pages:
            text += page.extract_text()
    return text

def lookup_entity(entity, kb_connection):
    cursor = kb_connection.cursor()
    cursor.execute("SELECT * FROM entities WHERE name LIKE ?", 
                   (f"%{entity}%",))
    return cursor.fetchall()

def process_document(pdf_path, kb_connection):
    text = extract_text(pdf_path)
    
    # Simple extraction (customize for your data)
    entities = {
        "company": extract_pattern(text, r'Company:\s*(.+?)(?=\n|$)'),
        "date": extract_pattern(text, r'\d{4}-\d{2}-\d{2}'),
    }
    
    # Enrich with KB
    enriched = {}
    for key, value in entities.items():
        kb_data = lookup_entity(value, kb_connection)
        enriched[key] = {"extracted": value, "kb_match": kb_data}
    
    return enriched

def extract_pattern(text, pattern):
    import re
    match = re.search(pattern, text)
    return match.group(1) if match else None
```

# Quality Assurance System for AI-Generated Content

This is a great use case for **model tiering + subagents**. Here's the architecture I'd recommend:

## Recommended Architecture

```
Input Content
    ↓
[Haiku] Fast Classification
    ├─ Language & format check
    ├─ Obvious errors filter
    ↓
[Sonnet] Parallel Reviews (3 subagents)
    ├─ Accuracy Reviewer
    ├─ Tone & Brand Reviewer
    └─ Compliance Reviewer
    ↓
[Opus] Final Synthesis & Decision
    ├─ Reconcile conflicts
    ├─ Generate remediation steps
    ├─ Make pass/fail decision
    ↓
Output (Pass/Fail + Report)
```

## Why This Pattern

- **Haiku upfront** (fast, cheap): Catches obvious issues early, filters out junk before expensive reviews
- **Parallel Sonnet subagents**: Each gets its own context + system prompt optimized for its specialty. They run independently, so you get 3 reviews simultaneously
- **Opus final review** (high reasoning): Reconciles conflicts, makes the final call, and generates actionable remediation if content fails

This keeps costs low while maintaining quality. You're not paying Opus to do basic checks.

## How They Work

**TCP (Transmission Control Protocol)**
- Establishes connection via 3-way handshake (SYN, SYN-ACK, ACK)
- Breaks data into segments, numbers them sequentially
- Receiver acknowledges receipt; sender retransmits lost packets
- Closes connection gracefully
- Best thought of as a "reliable pipe"

**UDP (User Datagram Protocol)**
- Sends data immediately without setup
- No acknowledgment or retransmission
- Minimal overhead—just wraps data in a header and sends it
- Best thought of as "fire and forget"

## When to Use Each

### **Use TCP for:**
- **Email (SMTP, POP3, IMAP)** — messages must arrive complete and intact
- **Web browsing (HTTP/HTTPS)** — pages, images, files must download fully
- **File transfer (FTP, SFTP)** — every byte matters
- **Remote access (SSH, Telnet)** — commands must execute reliably
- **Database connections** — transactions require certainty
- **Banking/financial apps** — accuracy is non-negotiable
- **Any scenario where packet loss is unacceptable**

### **Use UDP for:**
- **Video streaming (Netflix, YouTube)** — occasional dropped frames are acceptable; speed matters more
- **Voice over IP (Skype, Discord)** — low latency is critical; one lost audio sample isn't noticed
- **Online gaming** — responsiveness beats perfection; a missed position update is less bad than lag
- **Live sports broadcasts** — real-time delivery matters more than catching every frame
- **DNS queries** — can retry if no response; speed is important
- **IoT sensor data** — high volume, individual readings less critical
- **Network monitoring (SNMP)** — lightweight polling of many devices

## Practical Example

**Downloading a document (TCP):**
- You initiate connection
- Server confirms
- File transfers in order
- Missing pieces are resent
- Connection closes
- You have the exact file

**Playing an online game (UDP):**
- Your position updates sent continuously
- Some updates might not arrive—you see a slight hiccup
- No retransmission (would cause unacceptable lag)
- Game predicts where others are based on last-known position
- Low latency enables real-time interaction

## Hybrid Approach

Modern applications sometimes use **both**:
- **QUIC** (Google's protocol) combines TCP-like reliability with UDP's speed
- **Streaming apps** use UDP for video but TCP for setup, authentication, and buffering logic
- **Games** might use TCP for login/transactions, UDP for gameplay

- **TCP = Telephone call** — you connect first, ensure clarity, hang up properly
- **UDP = Shouting across a room** — you just yell; if someone doesn't hear, so be it

The choice comes down to one question: *Is getting the data there completely and in order more important than getting it there fast?* If yes → TCP. If speed matters more → UDP.
