# Document Analysis Pipeline Architecture

For a small team processing 100-page PDFs with cross-referencing, here's a straightforward approach:

## Recommended Architecture

**Single-pass processing with structured outputs** is your best bet:

```
PDF → Claude (1M context) → JSON summary → Knowledge base lookup → Final output
```

### Why This Works

1. **100-page PDFs fit in one pass** — At ~200K tokens of context available (with 1M beta), a 100-page PDF (~50-80K tokens) leaves plenty of room for your extraction schema and instructions. No chunking needed.

2. **Structured outputs guarantee valid JSON** — Use `output_config.format` with a schema matching your summary structure. No parsing failures.

3. **Knowledge base lookup happens after** — Once Claude extracts structured data, you do the cross-referencing in your own code (cheaper, faster, more control).

### Code Example

```python

# Your extraction schema

    "type": "json_schema",
    "schema": {
        "type": "object",

            "document_title": {"type": "string"},
            "key_entities": {
                "type": "array",
                "items": {"type": "string"}
            },
            "important_dates": {
                "type": "array",
                "items": {"type": "object", "properties": {
                    "date": {"type": "string"},

                }}
            },
            "financial_figures": {
                "type": "array",
                "items": {"type": "object", "properties": {
                    "amount": {"type": "string"},
                    "description": {"type": "string"}
                }}
            },

            "action_items": {"type": "array", "items": {"type": "string"}}
        },

            "content": [
                {
                    "type": "text",
                    "text": "Extract key information from this document. Focus on: entities, dates, financial figures, and action items."
                },
                {
                    "type": "document",
                    "source": {
                        "type": "base64",

                        "data": pdf_base64
```

```python
def cross_reference_knowledge_base(extracted_data, kb):
    """Match extracted entities against your KB"""
    refs = {}
    
    for entity in extracted_data.get("key_entities", []):
        # Simple lookup (adapt to your KB structure)
        if entity in kb:
            refs[entity] = kb[entity]
    
    return {
        "extracted": extracted_data,
        "cross_references": refs
    }
```

## How Streaming Works with Tools

When you enable `stream: true`, Claude streams **content blocks progressively** as they're generated. Here's the key insight:

**Tool calls stream differently than text:**

The flow looks like:
```
User message
    ↓
Stream starts → [message_start] → [content_block_start] → [content_block_delta] ... → [content_block_stop]
                                                                ↑

    ↓
Tool call complete → You execute tool → Stream resumes with assistant's next message
```

## Implementation Pattern

Here's a complete example with web search and code execution:

```typescript

    id: string;
    name: string;
    input: Record<string, unknown>;
  } | null = null;
  let accumulatedInput = "";

  const toolUseStream = await client.messages.stream({
    model: "claude-opus-4-6",

      {
        type: "builtin_tool",

        role: "user",

    ],

  // Process stream events
  for await (const event of toolUseStream) {
    switch (event.type) {
      case "content_block_start":
        if (event.content_block.type === "tool_use") {
          currentToolCall = {
            id: event.content_block.id,

          accumulatedInput += event.delta.partial_json;
```

# TCP vs UDP

## Key Differences

| Feature | TCP | UDP |
|---------|-----|-----|
| **Connection** | Connection-oriented | Connectionless |
| **Reliability** | Guaranteed delivery | Best-effort (no guarantees) |
| **Ordering** | Maintains packet order | No ordering guarantee |
| **Speed** | Slower (more overhead) | Faster (minimal overhead) |
| **Error checking** | Extensive | Minimal |
| **Header size** | 20+ bytes | 8 bytes |

## How They Work

**TCP (Transmission Control Protocol)**
- Establishes connection before sending data
- Confirms receipt of each packet
- Resends lost packets automatically
- Like a phone call—connection is established first

**UDP (User Datagram Protocol)**
- Sends data immediately without setup
- No confirmation of delivery
- Faster but unreliable
- Like sending postcards—fire and forget

## When to Use Each

### Use **TCP** when:
- Accuracy is critical
- Email, file transfers, web browsing (HTTP/HTTPS)
- Financial transactions
- Remote access (SSH, Telnet)
- You can tolerate slight delays

### Use **UDP** when:
- Speed matters more than perfection
- Video/audio streaming (can handle small losses)
- Online gaming (low latency is crucial)
- Live broadcasting
- DNS queries
- IoT sensor data
- Losing a few packets is acceptable

# TCP vs UDP

These are the two primary transport layer protocols used for sending data over networks. Here are the key differences:

## **TCP (Transmission Control Protocol)**

**Characteristics:**
- **Connection-oriented** — establishes a connection before sending data (three-way handshake)
- **Reliable delivery** — guarantees all packets arrive in order, with automatic retransmission of lost packets
- **Ordered** — data arrives in the sequence it was sent
- **Flow control** — manages transmission rate to prevent overwhelming the receiver
- **Error checking** — detects and corrects transmission errors
- **Slower** — overhead from reliability mechanisms

**How it works:**
1. Sender initiates connection
2. Data is broken into segments
3. Each segment is tracked and acknowledged
4. Lost segments are resent

## **UDP (User Datagram Protocol)**

**Characteristics:**
- **Connectionless** — sends data immediately without establishing a connection
- **Unreliable delivery** — no guarantee packets arrive; no retransmission
- **Unordered** — packets may arrive out of sequence
- **No flow control** — sends at application's pace
- **Minimal error checking** — only detects errors, doesn't correct them
- **Faster** — minimal overhead

**How it works:**
- Application sends data packets directly
- Packets are sent independently with no tracking
- Lost packets are simply lost

## **When to Use Each**

| Use Case | Protocol | Why |
|----------|----------|-----|
| **Email, file transfer, web browsing** | TCP | Data integrity critical; acceptable if slightly slower |
| **Video/audio streaming** | UDP | Speed matters more than perfection; occasional lost frame acceptable |
| **Online gaming** | UDP | Low latency essential; players can handle occasional glitches |
| **VoIP calls** | UDP | Real-time communication; slight audio loss acceptable |
| **DNS lookups** | UDP | Simple queries; can retry if needed |
| **Database transactions** | TCP | Accuracy paramount; must guarantee data consistency |
| **Remote desktop/SSH** | TCP | Need reliability for command execution |
| **Live video conferencing** | UDP | Speed prioritized over minor quality loss |
| **IoT sensors** | Often UDP | High volume, some data loss acceptable |

## **Quick Decision Framework**

- **Choose TCP** if data loss is unacceptable (financial data, messages, files)
- **Choose UDP** if speed and low latency matter more than perfect delivery (real-time media, gaming)
